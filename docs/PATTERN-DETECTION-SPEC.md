# Pattern Detection Specification

> **CRITICAL:** This defines the EXACT pattern-detection logic for git-slot-machine. The
> **payouts and odds** live in the canonical `patterns.json`; this document describes the
> detection **algorithm** that the TypeScript (CLI), PHP (server), and browser JS
> implementations MUST all follow identically. A cross-language **golden-vector contract
> test** enforces agreement — any drift fails CI.

## Version

**Ruleset / spec version:** 3.0.0
**Source of truth for payouts & odds:** `patterns.json` (identical copy in both repos)
**Ruleset content hash:** `9fc011bad925` (exposed at `GET /api/ruleset-version`)

## Source of Truth & Sync

- `patterns.json` is **generated by exhaustive enumeration** of all 16⁷ hashes (see
  `git-slot-machine/scripts/verify-odds.mjs`) — every `net` count and `oneIn` value is exact,
  not estimated.
- **PHP** (`app/Services/PatternDetector.php`) loads payouts from `resources/data/patterns.json`
  at runtime. **TS** (`src/patterns.ts`) keeps an in-code `PAYOUTS` mirror guarded by a Jest
  contract test (`patterns.contract.test.ts`). The **odds page** renders entirely from the JSON.
- The CLI can compare its bundled `rulesetHash` to `GET /api/ruleset-version` (throttled,
  fail-open) to detect when its payout table has drifted from the server's.

## Input Validation (unchanged)

1. Length MUST be exactly 7 → else throw `"Hash must be 7 characters"`.
2. Hex only (`0-9a-fA-F`) → else throw `"Hash must contain only hex characters"`.
3. Lowercase the hash AFTER validation, BEFORE detection.

## Detection Order — rarest first, FIRST MATCH WINS

> **Note:** the *detection priority* below is NOT the same as the *payout ladder*. Detection
> order is what makes the algorithm decidable; payouts follow true rarity (see Odds). For
> example THREE PAIR is checked late (priority 14) but is genuinely rare and pays 1,000.

`distribution` = character counts sorted descending. `consecutivePairs` = adjacent equal chars
counted greedily left-to-right, skipping 2 on a match. `sequentialRun(n)` = a contiguous window
of `n` positions whose hex values step by exactly ±1, all ascending OR all descending.

## The Tricky Rules — READ THIS (these are the non-obvious ones)

1. **Pairs are ADJACENCY-based, not frequency-based.** A "pair" requires the two identical
   characters to be in *neighbouring positions*. So `a1a2345` (two a's, not adjacent) is
   **NO_WIN**, and `ab1ab2c` (a,b each twice, none adjacent) is **NO_WIN** — not two pair.
2. **Straights are contiguous positions, step exactly ±1, ascending XOR descending, no
   wraparound.** `abcdefa` is a **BIG STRAIGHT** (positions 0–5 = `abcdef`), *not* ALPHABET
   SOUP. Scrambled-but-present digits do not count.
3. **ALL_LETTERS ranks HIGH (priority 8, above 5-straights and all houses); ALL_NUMBERS ranks
   LOW (priority 16, below two pair).** Intentional — 6 letter-symbols are rarer than 10 digit-
   symbols. Consequence: `aaabbcc` → **ALPHABET SOUP**, and `1234567` → **LUCKY SEVEN** (not
   ALL NUMBERS). Many all-digit hashes get classified as something higher.
4. **FULLER_HOUSE (3-2-2) is checked BEFORE FULL_HOUSE (3-2-1-1)** and pays differently
   (250 vs 25). Both satisfy `distribution[0]===3 && distribution[1]===2`; FULLER additionally
   requires `distribution[2]===2`.
5. **`7777777` → LUCKY SEVENS** (the secret), not JACKPOT.

## Odds

Exact, computed by enumerating all **16⁷ = 268,435,456** hashes through the detector. Every
pattern's `net` count, `oneIn`, `probability`, and a verified closed-form `formulaLatex` are in
`patterns.json`. The detection priority *absorbs* overlaps: a hash that matches several patterns
counts only toward the rarest (highest-priority) one — so e.g. ALL_NUMBERS' raw 10⁷ shrinks to a
net 7,067,912 because straights/of-a-kinds/houses outrank it.

- **Win rate:** 36.69% (NO_WIN 63.31%).
- **RTP:** 109.09% (avg return 10.91 on a 10-point ante).

- **TS (CLI):** `git-slot-machine/src/patterns.ts` + `src/patterns.json`;
  `src/patterns.contract.test.ts` (golden + canonical + tricky + example validation);
  `scripts/verify-odds.mjs` (exhaustive math lock: enumerates 16⁷, asserts every `net` + RTP).
- **PHP (server):** `app/Services/PatternDetector.php` (reads `resources/data/patterns.json`);
  `tests/Unit/PatternDetectorTest.php`, `tests/Unit/PatternContractTest.php`,
  `tests/Feature/OddsPageTest.php`.
- **Browser:** `resources/js/patterns.js` (classification only; payouts come from the server).
- **Shared contract:** `tests/Fixtures/golden-vectors.json` / `src/__fixtures__/golden-vectors.json`
  (547 vectors, all 19 types) asserted by both suites — this is what prevents cross-language drift.

## Sync Protocol

1. Change payouts/order ONLY by regenerating `patterns.json` (do not hand-edit the numbers).
2. Copy the regenerated `patterns.json` into both repos.
3. Run `pnpm verify:odds` (TS) — asserts net counts + RTP against the JSON.
4. Run both test suites: `pnpm test` (jest) and `php artisan test` (pest).
5. The `rulesetHash` changes automatically; bump `rulesetVersion` for a human-facing version.
6. Do NOT deploy one implementation without the other.
